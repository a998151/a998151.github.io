Java初认识锁机制

锁的概念

在介绍锁的概念之前，先了解一下并发的概念。

举一个例子来说，一个厨师在正常情况下我们不允许厨师同时做两道菜，先做完一道再做另一道。但是如果时间紧迫的情况下（特定情况下），如果希望能同时拿出两道菜，如果按照之前的情况来看，明显完成不了任务，但是我们允许厨师可以做着其中一道菜同时兼顾着其他菜，那就有可能完成任务。这里的厨师能做一道菜同时兼顾其他菜，我们就可以成为“并发”了。

我对并发的理解就是：对于某个任务，我们能够并行执行多个部分去共同完成这个任务，已达到提高我们完成工作效率效果，这就算作并发。

有了并发就可以同时做多件事情，多个线程彼此之间的影响的问题也就随着出现了。

对于并发的工作，需要某种方式来防止两个或多个任务访问相同的资源（至少在关键时候不可以），防止这种情况发生的方法就是资源被第一个调用者调用的时候，其它的调用着不能再调用该资源，其它需要调用该资源的调用者们，需要进行等待（阻塞状态），在当前资源调用者说明它已经使用完毕该资源，其它调用者才可以进行调用资源。这里在第一个调用者调用资源 的时候我们为 其加上 “锁”，其它调用者就知道资源给调用了，进入等待状态。好比我们上厕所，如果厕所们关着，说明有人在使用。

使用锁的时机

可以运用 Brian 的同步规则：

如果你正在写一个变量，它可能接下来将被另一个线程读取，或者正在读取一个上一次已经被另一个线程写过的变量，那么你必须使用同步，并且，读写线程都必须使用相同的监视器锁同步。

注意：
我们使用锁，锁的是我们这个项目程序的资源，而不是数据库资源，另外，如果程序是分布式的，要特别注意资源是否需要分布式同步。

synchronized

Java 提供了synchronized 关键字，在程序要执行到 synchronized 保护的代码时候，它会先检查锁是否可用，确定锁可用然后获取锁，锁不可用则会进行等待其它线程调用完该代码后释放锁，再获取锁。执行完保护的代码片段完后再释放锁。（通过 synchronized 获得的锁为悲观锁）

synchronized 和 static synchronized

synchronized 是对类的当前实例进行加锁，防止其他线程同时访问该类的该实例的所有 synchronized 块，注意这里是“类的当前实例”，类的两个不同实例就没有这种约束了。

static synchronized 恰好就是要控制类的所有实例的访问了，static synchronized 是限制线程同时访问 JVM中该类的所有实例同时访问对应的代码块。

实际上，在类中如果某方法或某代码块中有 synchronized，那么在生成一个该类实例后，该实例也就有一个监视块，防止线程并发访问该实例的 synchronized 保护块，而 static synchronized 则是所有该类的所有实例公用得一个监视块，这就是他们两个的区别。
