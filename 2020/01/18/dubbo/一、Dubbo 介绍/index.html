在介绍 Dubbo 分布式框架

RPC 介绍

RPC 是什么？

推荐一个回答：谁能用通俗的语言解释一下什么是 RPC 框架？

RPC 是通过网络的通信数据来表达要调用对方的某些功能。本地需要调用服务端的方法通过 client stub 寻址，然后序列化参数，通过 sockets 封装数据发送服务端的 sockets ，服务端接收到了数据，进行反序列化参数，然后调用对应的方法执行；服务端有了执行的结果后，通过 server stub 记录的客户端访问的地址，进行响应给调用服务的客户端。

  • RPC目标
    • 实现远程的接口（功能方法）的调用
    • 不损失本地调用的语义简洁性（
  • RPC 两个关键点
    • 寻址
    • 序列化

HTTP 和 RPC 区别？

这里推荐两篇文章：

既然有 HTTP 请求，为什么还要用 RPC 调用？

Http和RPC区别

从广度上来讲的话，我个人觉得 HTTP 算是 RPC 的一种实现。

公司A，可能实现自己的 RPC 调用框架和公司B实现的 RPC 调用框架不一样，遵循的规则不同，则会出现不兼容的情况，两者不可进行直接通信。如果两公司都是通过 HTTP 来实现接口，那可以进行 HTTP 网络通信。

首先HTTP 和 RPC 不是同一个层次上的技术或规范，RPC 的功能是实现远程调用，这个调用通常包含 传输协议 和 序列化协议。

  • 在 dubbo 中，实现 RPC 调用的传输协议是自定义报文的 TCP 协议。
  • 序列化协议中，目前流行的 json 和 xml，也有二进制编码的 protobuf hessian等。

建议

  • 接口开放给第三方调用，则用 HTTP 的 RESTful 风格接口
  • 接口用于内部系统使用，则使用 RPC 架构

RPC框架的好处

  • 首先就是长链接，不必每次通信都要像http一样去3次握手什么的，减少了网络开销；
  • 其次就是RPC框架一般都有注册中心，有丰富的监控管理；
  • 发布、下线接口、动态扩展等，对调用方来说是无感知、统一化的操作。

使用场景

RPC主要是用在大型企业里面，因为大型企业里面系统繁多，业务线复杂，而且效率优势非常重要的一块，这个时候RPC的优势就比较明显了。

基本架构图

我们先来看一下 Apache Dubbo 框架的功能实现图。

可以从途中看出有三种线条分别代表着初始化、异步调用、同步调用的意思。

1、容器中的服务启动的时候，会先注册到注册中心

2、消费者会向注册中心订阅服务

3、有新的服务发布，注册中心通知消费者

4、消费者进行同步调用服务

5、再消费则进行调用或者生产者工作的时候，监控者会进行监听生产者或者工作者的业务工作
