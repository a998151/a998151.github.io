缓冲区

​ 缓冲区是我们在前面所讨论的，用来暂存、中转数据的一块内存区域，在 Java NIO 中对此抽象出来用类表示，也就是 Buffer 类。对于每个非布尔原始数据类型都有一个缓冲区类。

​ 缓冲区的工作与通道紧密联系。通道是 I/O 传输发生时通过的入口，而缓冲区是这些数据传输的来源或目标。有关程序读写数据，数据在Buffer 与 Channel 中传输的流程如下图所示。

Buffer 属性

先来看一下 Java NIO 中的 Buffer 有哪些属性，其中四个基本的属性

mark

​ 标记一个位置。调用 mark() 来设定 mark = position。调用 reset() 设定 position = mark。通俗点说，我用 mark() 方法将在数据某处打上一个标志，之后调用 reset() 方法后，读写数据时候，数据将会从我标记的地方开始进行读写。

注：标记在设定前是未定义的。

position

​ 下一个要被读或写的元素的索引（指针）。位置会自动由相应的 get() 和 put

limit

​ 缓冲区现存元素的计数。limit 后面的数据不可进行读写。

capacity

​ 缓冲区能够容纳的数据元素的最大数量。这一容量在缓冲区创建时设定，并且永远不能被改变。

基本属性之间的关系

​ 如上图所示，给数据的第1个元素打上 mark，当时读/写操作到了 第3个元素上，这次的读/写数据量有 9 个，这个 Buffer 最大容量是 12。

Buffer API

方法 说明
public final int capacity() 获取Buffer容量
public final int position() 获取指针位置
public final Buffer position(int newPosition) 设置 Buffer 的 position
public final int limit() 返回 Buffer 的上界
public final Buffer limit(int newLimit) 设置 Buffer 的上界
public final Buffer mark() 将 mark 设置为 position
public final Buffer reset() 将 Buffer 的 position 重置为先前的 mark
public final Buffer clear() 清空（重置）缓冲区，但是里面的数据还是存在，仅是基本属性发生了改变
public final Buffer flip() 将写模式翻转为读模式，看源码就知道它是怎么弄的了。
public final Buffer rewind() 将当前位置置0，并同时取消 mark 标记
public final int remaining() 当前位置到上界还剩的元素数目
public final boolean hasRemaining() 判断是否还有剩余元素
public abstract boolean isReadOnly() 是否是只读
public abstract boolean hasArray(); 判断这个缓冲区是否被一个可访问数组支持
public abstract Object array(); 返回支持的数组
final int nextGetIndex(int nb)、nextPutIndex(int nb) 增加指定值的 position

以上的方法具体作用都可以通过源码中的文档进行阅读，比较简单。

​ 对于 isReadOnly() 方法，Buffer 都是可读的，但并不是所有 Buffer 都是可写的，对于一些特定的 Buffer ，可以进行重写 isReadOnly 方法进行实现。

对于特别的方法说明：

flip

​ 如果我们已经写满了缓冲区，现在我们准备将其清空。我们想把这个缓冲区传递给一个通道，以使内容能被全部写出。但如果通道现在在缓冲区上执行 get() ，那么它将从我们刚刚插入的有用数据之外取出未定义数据。如果我们将位置值重新设为 0 ，通道就会从正确位置开始获取

但是它是怎样知道何时达到我们插入数据末端的呢？

​ 这就是上界属性被引入的目的。上界属性指明了缓冲区有效内容的末端，我们需要将上界属性设置为当前位置，然后将位置重置为 0。

​ 但这种从填充到释放状态的缓冲区翻转是 API 设计者预先设计好的，它们为我们提供了一个非常遍历的函数

​ flip 函数将一个能够继续添加数据元素的填充状态的缓冲区翻转成一个准备读出元素的释放状态。

直接缓冲区和非直接缓冲区

​ 在Java中当我们要对数据进行更底层的操作时，通常是操作数据的字节（byte）形式，这时常常会用到ByteBuffer这样一个类。ByteBuffer提供了两种静态实例方式：

为什么提供这两种方式呢？

因为他们分配内存的方式并不一样。

直接缓冲区： 通过 allocateDirect() 方法分配，将缓冲区建立在物理内存中。可提高效率。
非直接缓冲区： 通过 allocate() 方法分配，将缓冲区建立在 JVM 的内存中。

为什么建立在物理内存中的就可以提高效率呢？

​ 外部数据提供给 Java 程序使用的过程是，数据先进入到系统的内存中，然后再通过内存的拷贝到 JVM 内存提供给 Java 程序使用。所以试用 allocateDirect()可以省去内存拷贝这一步骤，提高效率。

​ 但是系统级内存的分配比起 JVM 内存的分配要耗时得多（系统内存分配时间可能 > JVM 内存分配时间 + 内存数据拷贝时间），所以并不是任何时候 allocateDirect()的操作效率都高。

下面是一个不同容量情况下两种分配方式的操作时间对比：

​ 由图可以看出，当操作数据量很小时，两种分配方式操作使用时间基本是相同的，第一种方式有时可能会更快，但是当数据量很大时，第二种方式会远远大于第一种的分配方式。
